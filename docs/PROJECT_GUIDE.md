# PROJECT_GUIDE (프로젝트 개발 가이드)

본 문서는 **HWP Batch Printer** 프로젝트의 아키텍처 설계와 개발 원칙, 그리고 프로젝트의 의사결정 이력을 보존하는 프로젝트 특화 가이드입니다. 

---

## 1. 아키텍처 및 레이어 구조

### 1-1. 기술 스택 및 선택 이유
- **언어**: Python 3.11+ (빠른 개발 및 풍부한 에코시스템 활용)
- **GUI**: PySide6 (Qt6) (안정적인 Windows 네이티브 UI 제공)
- **COM**: Hancom Office COM (pywin32) (실제 시스템에 설치된 한글 엔진을 직접 활용하여 정확한 렌더링 보장)
- **배포**: PyInstaller (의존 라이브러리를 포함한 단일 exe 독립 실행 파일 빌드)
- **경로**: pathlib (OS 독립적이고 안전한 파일 경로 조작)
- **스레드**: QThread (COM 제어 및 시간이 걸리는 인쇄/PDF 변환 작업을 UI 스레드와 분리하여 응답성 확보)

### 1-2. 5-Layer 레이어 구조
관심사 분리(Separation of Concerns, SoC)와 단일 책임 원칙(SRP)을 강화하기 위해 다음의 5계층 아키텍처를 강제합니다.

```text
┌─────────────────────────────────────────┐
│              UI Layer                   │
│  main_window / drop_zone / file_list    │
│  control_panel / progress_panel         │
│  (화면에 보여지는 렌더링 및 입력 수신 담당)  │
├─────────────────────────────────────────┤
│            Service Layer                │
│  file_manager / hwp_service             │
│  pdf_service / print_service            │
│  log_service                            │
│  (비즈니스 로직 처리)                    │
├─────────────────────────────────────────┤
│            Worker Layer                 │
│  pdf_worker / print_worker              │
│  (QThread 기반 백그라운드 작업 처리)        │
├─────────────────────────────────────────┤
│             Core Layer                  │
│  app_state / models / constants         │
│  signals                                │
│  (전역 데이터, 상태, 설정, 공통 데이터 모델)   │
├─────────────────────────────────────────┤
│            Utility Layer                │
│  file_utils / com_utils                 │
│  (시스템, 파일시스템, COM 객체 관리 공통 헬퍼)│
└─────────────────────────────────────────┘
```

- **UI Layer**: 화면 렌더링 및 UI 위젯 표시만 담당합니다.
- **Service Layer**: 파일 관리, 한글 자동화, PDF 변환, 인쇄 및 로깅 처리를 담당합니다.
- **Worker Layer**: UI 프리징 방지를 위한 QThread 백그라운드 태스크입니다.
- **Core Layer**: AppState 싱글톤 모델, 상숫값 및 전역 시그널을 제공합니다.
- **Utility Layer**: 파일 경로 유틸리티 및 win32com 관련 폴백 로직을 제공합니다.

### 1-3. 상태 관리 및 통지 구조
- **AppState (Singleton)**: 모든 공유 상태(등록된 파일 목록, 처리 진행율, 옵션 상태 등)의 단일 진입점입니다.
- **상태 전파**: 상태 변경 시 Qt Signal을 통해 UI 및 필요한 계층에 이벤트를 자동으로 통지합니다.
- **금지 사항**: UI 프레임(`Frame`) 간에 서로의 상태를 직접 수정하거나 공유하는 행위를 절대 금지합니다. Worker Thread에서 UI 요소를 직접 수정하는 것 역시 금지되며, 반드시 Signal/Slot 메커니즘을 이용해야 합니다.

## 2. 핵심 개발 및 안정성 원칙

### 2-1. Hancom COM 안정성 원칙
한글 COM(Component Object Model)은 외부 프로세스로 구동되어 백그라운드에서 좀비 프로세스로 남기 쉽고, 동시성 오류에 매우 취약합니다.
1. **명시적 수명 관리**: 모든 COM 객체는 반드시 생성 및 해제를 try/finally 블록을 통해 명시적으로 수행해야 합니다. 작업 완료 또는 예외 발생 시 반드시 `Quit()` 메서드를 호출하여 백그라운드 프로세스가 남지 않도록 합니다.
   ```python
   hwp = None
   try:
       hwp = win32.gencache.EnsureDispatch("HWPFrame.HwpObject")
       # 비즈니스 로직 수행
   finally:
       if hwp:
           hwp.Quit()
   ```
2. **병렬 COM 작업 금지**: HWP COM은 멀티스레드 상에서 동시에 돌리면 비정상 종료를 유발할 확률이 극도로 높습니다. 따라서 **큐(Queue) 기반의 완전한 순차 처리(Sequential Queue)**를 강제합니다.
3. **접근 스레드 제한**: COM 작업은 UI Thread 상에서 호출할 경우 윈도우 먹통을 발생시키므로, 반드시 별도로 분리된 Worker Thread 내부에서만 작동해야 합니다.

### 2-2. GUI 및 스레드 응답성 원칙
1. **UI Freezing 금지**: 파일 로딩, 한글 문서 열기, PDF 변환 및 실제 인쇄와 같이 0.5초 이상 지연될 수 있는 연산은 메인 UI 스레드에서 즉시 비즈니스 연산을 호출하지 않고, `QThread`를 상속받은 백그라운드 Worker로 실행해야 합니다.
2. **UI 갱신 격리**: Worker Thread는 UI 컴포넌트를 직접 참조하거나 멤버 함수를 다이렉트로 호출할 수 없으며, 사전에 연결된 Signal을 Emit하여 UI 스레드가 수신하도록 합니다.
   ```python
   # 올바른 예시
   worker.progress_changed.connect(self.update_progress)
   ```

### 2-3. 파일 무결성 및 저장소 정책
1. **원본 파일 보존**: 사용자가 올린 원본 HWP/HWPX 파일은 읽기 전용(Read-Only)으로만 열며, 프로그램 내에서 내용을 절대 수정하지 않습니다.
2. **안전 저장 (Safe Save) 프로세스**: PDF 변환 또는 임시 생성 시, 즉시 대상 경로에 덮어쓰지 않고 임시 폴더나 temp 파일 명으로 저장한 후, 저장이 정상 완료된 것을 검증한 후 실제 목적지 파일명으로 교체(`rename`)합니다.
3. **출력 폴더 표준**: 변환된 PDF 결과물은 항상 원본 폴더 하위에 `PDF_OUTPUT/` 디렉토리를 자동 생성(`mkdir(exist_ok=True)`)하여 한곳에 관리되도록 합니다.

### 2-4. 작업 큐 및 예외 극복 (Recovery)
1. **에러 격리**: 대량 파일 일괄 처리 시, 특정 파일 하나가 포맷 오류나 보안 문제 등으로 처리에 실패하더라도, 전체 배치 처리가 중단되어서는 안 됩니다. 해당 파일에 대한 오류 로그 및 상태를 기록하고 즉시 다음 대기 파일 처리를 재개합니다.
2. **복구 절차**: 예외가 감지되면 오류 로그 및 스택트레이스를 기록하고, 즉시 열려 있는 HWP 인스턴스를 소멸시킨 후 Worker를 정상 안전 종료하고 UI는 대기 상태를 유지하여 다음 대기 큐를 처리하거나 사용자가 다시 시작할 수 있도록 제어를 반환합니다.

### 2-5. 로깅 체계
시스템 내부에서 발생하는 세부 내역은 분석이 가능하도록 logs 폴더 산하에 분리하여 저장합니다.
- `logs/app.log`: 일반 프로그램 생명주기 및 사용자 액션 로그
- `logs/convert.log`: PDF 변환 성공/실패 기록
- `logs/print.log`: 인쇄 명령 송신 및 옵션 기록
- `logs/error.log`: 전체 레이어에서 발생한 예외 및 Traceback
* **필수 기록 항목**: 파일명, 시작 및 종료 시간, 작업 결과, 에러 메시지, Traceback, 적용된 프린터 하드웨어 설정 및 옵션

### 2-6. 데이터 구조 규칙
- 데이터 정의를 위해 Pydantic 등 무거운 프레임워크 사용을 지양하고, Python 표준 라이브러리인 `@dataclass`를 우선적으로 채택합니다.
- 공유 상태는 컨트롤러 혹은 `AppState` 싱글톤 클래스 내에서 단일 통제합니다.

## 3. 검증 시스템 및 배포 원칙

### 3-1. 위험도 기반 검증 기준 (Hell Validation)
수정 사항의 복잡도와 위험성에 따라 검증 기준을 3단계로 구분하여 적용합니다.
- **Level 1 (Fast Patch)**: 단순 UI 문구, 배경 색상, QSS 스타일 수정. 코드 리뷰 1회로 간소화 검증.
- **Level 2 (Standard)**: 전역 상태 변경, 파일 추가/제거 이벤트 처리, UI 흐름 제어 로직. 실제 실행 테스트 및 빈 데이터/비정상 파일 로딩 테스트 수행.
- **Level 3 (Hell Validation)**: HWP COM 제어 코드, 멀티스레드 동기화, 프린터 출력 인터페이스, PDF 생성 경로 변경 등. 대량 파일 테스트 및 예외 상황 강제 발생을 포함하여 최대 3회 이상 완벽히 교차 검증하며 롤백 계획을 명문화함.

### 3-2. 배포 및 패키징 원칙
- **PyInstaller**를 활용하여 의존 모듈이 포함된 단일 실행 파일(`.exe`)로 패키징합니다.
- 빌드 시, 스타일시트 등 리소스 누락을 미연에 방지하기 위해 `--add-data "assets;assets"` 설정을 누락 없이 지정하여 배포 무결성을 보장합니다.
- 한글 Windows의 CP949 콘솔 환경에서 유니코드 특수문자 출력으로 인한 인코딩 크래시를 막기 위해 빌드 스크립트의 모든 문구와 이모티콘을 표준 ASCII 영문 태그(`[OK]`, `[INFO]`)로 엄격하게 일원화하여 작성합니다.

## 4. 개발 금지 사항 및 최종 원칙

### 4-1. 개발 금지 사항 (Do Not)
- UI Thread에서 COM 객체 인스턴스 직접 호출
- HWP 문서를 병렬로 여는 행위 (병렬 처리)
- 원본 HWP/HWPX의 내용 변경 시도
- 로컬 경로나 환경 변수의 하드코딩
- 하나의 함수에 200줄 이상의 비대 코드 작성 (함수 분할 필수)
- 모든 역할을 한데 몰아넣은 God Object 클래스 설계
- try-except 없이 COM 객체를 선언하여 프로세스 누수를 유발하는 코드
- 로그 기록 없이 단순 패스(`pass`)하는 예외 처리

### 4-2. 최종 개발 지침
1. **빠른 구현보다 오래 지속될 안정적인 코드를 우선**하여 선택합니다.
2. COM 객체는 어떠한 실패 경로에서도 반드시 완벽하게 `Quit()`으로 소멸시킵니다.
3. UI는 어떠한 처리 부하에서도 멈추지 않고 항상 진행 상태 피드백을 유지해야 합니다.
4. 실패하더라도 프로그램이 멈추지 않고 다음 대기 작업을 처리할 수 있는 유연한 에러 복구를 추구합니다.

| 결정 사항 | 최종 선택 | 도입 목적 및 상세 근거 |
